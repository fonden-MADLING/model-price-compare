# 前端 UI 设计方案

## 设计目标

这个页面不是模型厂商大全，也不是营销型首页。第一版要解决三个具体问题：

1. 用户能快速筛选出自己关心的模型。
2. 用户能按真实使用场景计算价格，而不是只看静态单价。
3. 用户能看到每个关键价格字段来自哪份官方文档。

## 信息架构

第一版只做一个主页面：

```text
┌────────────────────────────────────────────────────────────────────┐
│ 顶部栏：AI 模型价格对比 / 更新时间 / 数据版本 / 数据下载           │
├────────────────────────────────────────────────────────────────────┤
│ 成本计算器：输入 token / 输出 token / 缓存命中率 / 币种口径        │
├───────────────┬────────────────────────────────────────────────────┤
│ 筛选栏        │ 模型对比表                                         │
│ - 厂商        │ - 模型名称                                         │
│ - 国内外      │ - 厂商                                             │
│ - 上下文窗口  │ - 上下文窗口                                       │
│ - 价格范围    │ - 未命中输入价                                     │
│ - 证据状态    │ - 命中缓存输入价                                   │
│               │ - 输出价                                           │
│               │ - 场景成本                                         │
│               │ - 证据状态                                         │
└───────────────┴────────────────────────────────────────────────────┘
右侧抽屉：模型详情 / 官方文档链接 / 字段级证据 / 抽取时间 / 复核状态
```

## 页面布局

### 顶部栏

顶部栏保持紧凑，展示：

- 页面名称：`AI 模型价格对比`
- 数据更新时间
- 数据版本或 Git commit
- `下载 JSON` 按钮
- `查看数据源` 按钮

这里不做大标题和宣传文案，因为用户进入页面后的主要任务是比较价格。

### 成本计算器

成本计算器放在表格上方，作为第一优先级工具。

字段：

- 输入 token 数
- 输出 token 数
- 缓存命中率
- 计价单位：默认每次调用成本，可切换为每月成本
- 月调用量：只有切换到每月成本时展示

计算公式：

```text
输入成本 =
输入 token / 1,000,000 ×
（缓存命中率 × 命中缓存输入价 + 未命中率 × 未命中输入价）

输出成本 =
输出 token / 1,000,000 × 输出价

总成本 = 输入成本 + 输出成本
```

为什么要把成本计算器放在表格上方：用户比较模型时，静态单价不一定能代表实际成本，缓存命中率和输出长度会显著改变排序结果。

### 左侧筛选栏

筛选项：

- 厂商：多选
- 国内外：单选或多选
- 上下文窗口：`200K+`、`256K`、`262K`、`1M`
- 价格区间：输入成本、输出成本、场景总成本
- 证据状态：待抽取、自动抽取、待复核、已复核

筛选栏不做成浮夸卡片，只用紧凑分组，便于反复操作。

### 主对比表

默认列：

| 列名 | 说明 |
| --- | --- |
| 模型名称 | 主识别字段 |
| 厂商 | 支持筛选和分组 |
| 国内外 | 帮用户判断访问与采购偏好 |
| 上下文窗口 | 同时展示 label 和 tokens |
| 未命中输入价 | 元 / 百万 token |
| 命中缓存输入价 | 元 / 百万 token |
| 输出价 | 元 / 百万 token |
| 场景成本 | 根据顶部计算器动态计算 |
| 证据状态 | 标记数据可信度 |

交互：

- 表头支持排序。
- 模型名称点击打开右侧详情抽屉。
- 价格为空时显示 `待抽取`，不要显示 `0`。
- 异常价格用轻量提示标记，例如缓存价高于未命中价。

### 右侧详情抽屉

点击模型后打开。

内容：

- 模型名称、厂商、官方文档链接
- 上下文窗口说明
- 价格字段明细
- 字段级证据
- 抽取时间
- 置信度
- 人工复核状态

证据展示方式：

```text
未命中输入价：1 元 / 百万 token
证据：官方文档短摘录……
来源：https://...
抽取时间：2026-06-25 22:24
置信度：0.91
```

为什么用抽屉而不是弹窗：用户经常需要一边看表格排序，一边核对某个模型证据，抽屉不会打断主表上下文。

## 视觉风格

定位是工作型工具，视觉应克制、清晰、密度适中。

建议：

- 背景：浅灰或白色。
- 主色：蓝色只用于链接、选中态和关键操作。
- 状态色：绿色表示已复核，黄色表示待复核，红色表示异常。
- 表格行高保持紧凑，但证据抽屉可以更宽松。
- 避免大面积渐变、装饰图形和营销式 hero。

## MVP 必做

第一版必须完成：

- 读取 `data/models.json`
- 展示模型对比表
- 厂商和国内外筛选
- 表头排序
- 输入 token、输出 token、缓存命中率计算
- 模型详情抽屉
- 展示官方文档链接和证据字段

## 第一版暂不做

- 登录系统
- 在线编辑数据
- 后端接口
- 数据库
- 实时抓取官方文档
- 用户收藏同步
- 多人审核流

这些能力都不适合放进静态站第一版，否则会把低成本方案变成复杂系统。

## 后续增强

- 价格变化趋势：读取 `data/snapshots/` 做历史折线。
- 厂商维度统计：按厂商展示平均输入价、输出价、上下文窗口。
- 场景模板：客服、代码生成、长文档分析、Agent 调用。
- 异常检测：价格变化超过阈值时标记待复核。
